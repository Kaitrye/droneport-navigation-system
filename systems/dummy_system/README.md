# Заготовка проекта для системы

## Базовые требования

БТ1. Система предполагает использование нескольких компонентов. 

БТ2. Взаимодействие между компонентам должно происходить только через брокер сообщений и проверяться монитором безопасности на соответствие политикам безопасности.

БТ3. Для каждой системы должны быть перечислены и проанализированы активы, сформулированы цели и предположения безопасности, определены уровни доверия всех доменов безопасности.

БТ4. Для проверки политик безопасности должны быть реализованы автоматизированные тесты.

БТ5. Код доверенных доменов безопасности должен быть покрыт тестами не менее чем на 60%.

БТ6. Код доверенных доменов безопасности, повышающих целостность данных, должен быть покрыт тестами не менее чем на 70%, *включая зависимости*. 
Все зависимости должны быть проанализированы на предмет известных уязвимостей. В доверенных доменах безопасности запрещается использовать сторонние пакеты с известными уязвимостями.

Оговорка: т.к. проект учебный, в отдельных случаях допускаются исключения для тестирования зависимостей, в таком случае должен быть проведён статический анализ кода, проанализирована информация в публичных банках данных угроз (например, БДУ ФСТЭК)

БТ7. Код остальных (недоверенных) доменов безопасности должен быть протестирован не менее чем на 30% (без учёта зависимостей).

БТ8. Доля доверенной кодовой базы не должна превышать 15% от общего количества кода.

БТ9. Взаимодействие с другими системами может быть реализовано также через брокер сообщений (БС), который может в учебных целях как совпадать с БС для взаимодействия между внутренними компонентами, но может также и отличаться.

## Примечания

В зависимости от назначения системы, могут быть реализованы дополнительные интерфейсы взаимодействия между сервисами, однако они должны быть проанализированы и реализованы с учётом согласованных целей и предположений безопасности.
